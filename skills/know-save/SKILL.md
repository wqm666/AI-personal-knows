# 保存知识到知识库

## 何时使用

当对话中出现以下任一情况时，**必须**调用此技能保存知识：
- 做出了技术决策并有明确理由
- 发现了踩坑经验或非显而易见的问题
- 讨论了业务规则、流程、约束
- 总结了架构设计或组件关系
- 用户明确要求"记住这个"、"保存知识"

## 必须执行的步骤

### 1. 从对话中提取知识

分析当前对话，提取值得保存的知识。每条知识必须包含：

- **title**: 简洁的标题（一句话概括）
- **content**: 详细内容，必须包含 WHY（为什么这么做）而不仅是 WHAT（做了什么）
- **tags**: 相关标签，逗号分隔
- **knowledge_type**: 从以下选一个：
  - `pitfall` — 踩坑经验，容易犯的错
  - `decision` — 技术决策及其理由
  - `business` — 业务规则、流程、约束
  - `architecture` — 架构设计、组件关系
  - `lesson` — 经验教训、最佳实践
  - `fact` — 技术事实、配置参数
  - `faq` — 常见问题及答案

### 2. 调用 MCP 工具保存

调用 `mcp__personal-know__note_save` 工具：

```
content: [提取的知识内容，要详细、有上下文]
title: [简洁标题]
tags: [标签1, 标签2, ...]
knowledge_type: [类型]
source: conversation
```

### 3. 告知用户结果

保存后告诉用户：
- 保存成功/失败
- 如果被去重（reinforce），说明已有类似知识
- 如果发现矛盾（supersede），说明替换了旧知识
- 提醒用户：新知识处于「待审核」状态，需要在 Web UI 审核通过后才会被搜索召回

## 知识质量要求

保存的知识**必须**满足：
1. 有明确的上下文，独立阅读能理解
2. 包含 WHY，不仅仅是结论
3. 具体而非泛泛，包含版本/环境/条件等限定
4. 不保存公开文档已有的标准用法

## 示例

用户说："我们决定用 Redis 做分布式锁而不是 ZooKeeper，因为运维团队更熟悉 Redis，而且我们的并发量不需要 ZK 的强一致性"

保存为：
```
title: 分布式锁选型：Redis 优于 ZooKeeper
content: 选择 Redis 实现分布式锁而非 ZooKeeper。原因：1) 运维团队对 Redis 更熟悉，降低维护成本；2) 当前并发量级不需要 ZK 提供的强一致性保证，Redis 的性能更适合场景。权衡：如果未来并发量大幅增长或需要强一致性，需要重新评估。
tags: Redis, 分布式锁, 架构决策, ZooKeeper
knowledge_type: decision
```
