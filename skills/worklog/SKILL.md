# 记录工作日志

## 何时使用

**主动触发**（不需要用户明确要求）：
- 对话中完成了一个明确的任务（修了 bug、实现了功能、做了优化）
- 做了代码 review、部署、配置变更等操作
- 讨论并确定了方案（算一个决策工作项）

**被动触发**（用户明确要求）：
- "记一下今天的工作"
- "worklog"
- "/worklog"

## 必须执行的步骤

### 1. 从对话中提取工作项

每个工作项需要：
- **content**: 做了什么（一句话，具体但简洁）
- **project**: 所属项目（如果能识别的话）
- **tags**: 相关标签
- **duration**: 大概花了多久（分钟，估算即可）

### 2. 调用 MCP 工具

调用 `mcp__personal-know__worklog_add`：

```
content: [做了什么]
project: [项目名]
tags: [标签1, 标签2]
duration: [耗时分钟数]
```

date 参数不传，默认记录为今天。

### 3. 告知用户

简短确认："已记录到工作日志。"

## 记录原则

- **记 WHAT 不记 HOW** — "修复了用户登录超时问题" 而不是 "在 auth.go 第 42 行把 timeout 从 5s 改成 30s"
- **一条一事** — 每个独立任务单独记一条
- **简洁具体** — "优化了首页加载速度，从 3s 降到 800ms" 比 "做了性能优化" 好
- **不重复** — 如果同一个任务在对话中出现多次，只记最终完成的那一次

## 示例

对话中用户修复了一个 CORS 跨域问题，花了大约 30 分钟：

```
content: 修复 CORS 跨域问题 — nginx 和后端双重设置导致浏览器报错，改为仅 nginx 层处理
project: API Gateway
tags: CORS, nginx, bug fix
duration: 30
```
