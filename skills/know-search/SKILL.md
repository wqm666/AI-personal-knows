# 搜索知识库

## 何时使用

**主动触发**（不需要用户明确要求）：
- 用户问技术决策相关问题（"应该用什么方案"、"怎么选型"）
- 用户问业务规则（"这个流程是怎样的"、"有什么限制"）
- 用户提到"之前"、"上次"、"以前"怎么处理的
- 涉及架构设计、系统边界的讨论
- 遇到报错或问题，可能有已记录的踩坑经验

**被动触发**（用户明确要求）：
- "搜一下知识库"、"查查有没有相关知识"
- "/know-search xxx"

## 必须执行的步骤

### 1. 构造搜索查询

从用户的问题中提取关键词，构造搜索查询。注意：
- 使用核心概念词，不要用完整句子
- 中英文关键词都试一下效果更好
- 如果第一次搜索没结果，换同义词重试一次

### 2. 调用 MCP 工具搜索

调用 `mcp__personal-know__note_search` 工具：

```
query: [搜索关键词]
limit: 5
```

### 3. 分析并使用搜索结果

搜索结果处理规则：
- **有结果**：将知识库中的信息融入回答，标注"根据知识库记录..."
- **有矛盾**：如果结果中有 conflict_group，优先使用 freshness=latest 的版本，提醒用户存在旧版本
- **无结果**：正常回答，不要告诉用户"知识库里没找到"（除非用户主动搜的）
- **结果的 confidence 字段**：如果置信度低（<60），提醒用户该知识可能需要验证

### 4. 标记有用的结果

如果搜索结果对回答有帮助，调用 `mcp__personal-know__note_feedback`：

```
item_id: [有用的知识条目 ID]
```

## 搜索策略

| 场景 | 搜索词示例 |
|------|-----------|
| 用户问"Redis 怎么做分布式锁" | `Redis 分布式锁` |
| 用户说"支付流程有什么坑" | `支付 踩坑` 或 `支付 pitfall` |
| 用户问"上次数据库选型怎么决定的" | `数据库 选型 决策` |
| 用户报错"连接超时" | `连接超时 timeout` |

## 重要

- 搜索是零成本的，宁可多搜一次也不要错过已有知识
- 不要在回答中暴露搜索过程的技术细节（如 ID、score）
- 如果知识库的记录和你的通用知识矛盾，**优先信任知识库**（因为它是用户/团队的特定实践）
